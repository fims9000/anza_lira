# ANZA-LIRA

Рабочий репозиторий по сегментации протяженных структур (сосуды/дороги) с вариантом `AZ-Thesis` и сравнением против `U-Net baseline`.

## Текущий фокус (2026-04-27)

1. **Статья 2 (EDM)**: собираем финальный, проверяемый пакет метрик + фигур.
2. **Статья 1 (IMSC)**: цифры в тексте держим как **целевые/рабочие**, пока не как финальный независимый benchmark.
3. **Статья 3**: начинаем после фиксации материалов статьи 2.

## Важное правило по метрикам

- В таблицы статей вставляем только те числа, у которых есть:
  - явный источник (`results/.../metrics.json` или `..._summary.md`);
  - понятный протокол (seed, epochs, split, threshold policy).
- Если результат временный (проба/гипотеза), помечаем как `internal protocol` или `draft`.

## Минимальная структура проекта

- `train.py` — основной запуск обучения/валидации.
- `models/azconv.py` — реализация слоя и логики AZ.
- `configs/` — конфиги датасетов и протоколов.
- `scripts/` — очереди запусков, сбор сводок, подготовка фигур.
- `results/` — итоговые метрики, сводки, статьи, визуализации.
- `logs/` — stdout/stderr длительных запусков.
- `tests/` — базовые проверки пайплайна и модели.

## Быстрый запуск

```powershell
python -m venv .venv
.venv\Scripts\Activate.ps1
pip install -r requirements.txt
python train.py --config configs/fives_benchmark.yaml --variants baseline,az_thesis
```

## Политика чистоты репозитория

- В корне и в `results/` держим только актуальные сводки/пакеты для статей.
- Исторические и тяжелые артефакты — в архивных подпапках (например `results/_archive_pre_latest_20260426_150548/`) или вне git.
- Перед коммитом проверяем, что новые `.md` файлы ссылаются на реальные пути метрик/фигур.
